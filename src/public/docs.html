Dlp Wrapper

A self-hosted REST API wrapping yt-dlp. Fetch metadata, resolve direct CDN URLs, or download MP4/MP3 from 1,750+ sites — drop it into a web app or a Discord bot over plain HTTP.

$ curl -H "x-api-key: $KEY" "$BASE/api/media/info?url=..."

Quick start

Every request goes through the base URL of wherever you've deployed this — locally that's http://localhost:8080, on Render it's the onrender.com URL your service gets.

# local
npm install
npm start
# → listening on http://localhost:8080

Every endpoint lives under /api. All routes below are relative to that.

Authentication

Set the API_KEY environment variable and every request must include it as a header:

x-api-key: your-secret-key
Left unset, the API is fully open. That's fine for poking at it locally, but set API_KEY before pointing a public URL at it. The key is checked with a constant-time comparison, and applies to every /media/* route — /healthz is always open.

GET /api/healthz

Liveness check. No auth required. Returns:

{ "status": "ok" }

GET /api/media/info

Full metadata for any yt-dlp-supported URL — title, thumbnail, duration, uploader, and every available format.

GET /api/media/direct-url

Resolves the underlying CDN URL(s) without routing any bytes through this server. Good for Discord embeds or an HTML5 <video src> — some CDNs will 403 without the original request headers, in which case fall back to /media/download.

ParamRequiredDefaultDescription
urlrequired—Source URL
formatoptionalbestyt-dlp format selector
curl -H "x-api-key: $KEY" \
  "$BASE/api/media/direct-url?url=https://vimeo.com/76979871&format=best"
{ "title": "...", "urls": ["https://cdn.example.com/..."] }

GET /api/media/download

Downloads via yt-dlp into a temp directory, then streams the file back as an attachment.

ParamRequiredDefaultDescription
urlrequired—Source URL
extoptionalmp4mp4 or mp3
formatoptionalbestvideo*+bestaudio/bestyt-dlp format selector. Only applies when ext=mp4 — mp3 always extracts bestaudio/best.
curl -H "x-api-key: $KEY" -OJ \
  "$BASE/api/media/download?url=https://vimeo.com/76979871&ext=mp3"

This endpoint has its own stricter rate limit and concurrency cap (see below) since each request spins up a real ffmpeg/yt-dlp process.

Rate limits & concurrency

Two layers protect the container from being overwhelmed: a request-rate limit per client IP, and a hard cap on concurrent yt-dlp/ffmpeg processes.

Env varDefaultApplies to
RATE_LIMIT_MAX / RATE_LIMIT_WINDOW_MS30 / 60000Whole API, per client IP
DOWNLOAD_RATE_LIMIT_MAX / DOWNLOAD_RATE_LIMIT_WINDOW_MS10 / 600000/media/download only
DLP_MAX_CONCURRENT_INFO8Concurrent /media/info + /media/direct-url
DLP_MAX_CONCURRENT_DOWNLOADS2Concurrent /media/download

By default, hitting the concurrency cap returns an immediate 429. Set QUEUE_ENABLED=true to wait for a free slot instead:

Env varDefaultDescription
QUEUE_ENABLEDfalseQueue instead of rejecting immediately
QUEUE_MAX_WAIT_MS15000Give up and return 429 after this long waiting
QUEUE_MAX_LENGTH50Reject immediately once this many requests are already waiting

Set REDIS_URL to back the rate limiter with Redis instead of in-memory counters — counts then survive restarts and are shared across multiple instances rather than reset-per-process.

Metadata caching

/media/info and /media/direct-url cache their responses, since the same url is often requested repeatedly (a Discord command run twice, a web app re-rendering). /media/download is never cached — it's a stream, not a storable value.

Env varDefaultNotes
CACHE_TTL_INFO_MS3000005 minutes
CACHE_TTL_DIRECT_URL_MS120000Shorter — CDN URLs carry short-lived expiry tokens
CACHE_MAX_ITEMS500In-memory fallback only; ignored when REDIS_URL is set

Without REDIS_URL, the cache is an in-memory map scoped to this process. With it, entries are shared across every instance.

Per-IP quotas

A longer-window ceiling per IP, on top of the short-window rate limit above — e.g. "50 downloads per day" rather than "10 per 10 minutes". Opt-in, since not every self-hoster wants a hard daily cap.

Env varDefaultNotes
QUOTA_ENABLEDfalseMaster switch for all three quotas below
QUOTA_WINDOW_MS8640000024 hours
QUOTA_MAX_INFO500Per IP, per window
QUOTA_MAX_DIRECT_URL500Per IP, per window
QUOTA_MAX_DOWNLOADS50Per IP, per window

An IP over quota gets 429 with a Retry-After header. Counts use REDIS_URL when set, otherwise an in-memory map.

Metrics

Prometheus-format metrics are served at GET /metrics: request counts and latency per route, active job gauges, cache hit/miss counts, and quota rejections — plus the usual process/CPU/memory metrics.

curl http://localhost:8080/metrics

Set METRICS_KEY to require an x-metrics-key header on scrape requests. Left unset, /metrics is open — fine if you're only scraping it over a private network.

Security notes

  • SSRF guard — every url is checked before it reaches yt-dlp: must be http/https, and must resolve to a public address (blocks loopback, private ranges, and cloud metadata endpoints).
  • Constant-time key check — API_KEY comparison doesn't leak timing information.
  • CORS is closed by default — no browser origin can call the API until you set ALLOWED_ORIGINS.
  • Temp files are swept — per-request download directories are cleaned up on completion, with a periodic sweep for anything left behind by a crash.
If a web app calls this API, don't ship API_KEY to client-side JS — proxy the request through your own backend instead. Anything sent to a browser is extractable.

YouTube & cookies

YouTube requires cookies from a real browser session on any cloud IP — this is a yt-dlp limitation, not specific to this API.

  1. Export cookies in Netscape format (e.g. via a browser extension like "Get cookies.txt LOCALLY")
  2. Mount the file into the container, e.g. at /secrets/cookies.txt
  3. Set YTDLP_COOKIES_FILE=/secrets/cookies.txt

From there, cookie handling is managed automatically:

  • Writable working copy — the file is copied to a writable temp path at startup. yt-dlp rewrites its cookie jar after every run as tokens rotate, which silently fails against a read-only mount (e.g. Render Secret Files) — working from a copy keeps that refresh working.
  • Re-sync on change — the source file is re-checked on an interval (COOKIE_SYNC_INTERVAL_MS, default 5 minutes); dropping in a freshly exported file picks it up without a restart.
  • Expiry warnings — the cookie jar's expiry timestamps are parsed and logged as a warning when entries are expired or expiring within COOKIE_EXPIRY_WARNING_MS (default 24h).

That cookie session is shared by every caller of the API — treat it accordingly. None of this logs into anything on your behalf; you still export cookies from a real logged-in browser yourself.

curl

BASE="https://your-app.onrender.com/api"
KEY="your-secret-key"

curl -H "x-api-key: $KEY" "$BASE/media/info?url=https://vimeo.com/76979871"
curl -H "x-api-key: $KEY" "$BASE/media/direct-url?url=https://vimeo.com/76979871"
curl -H "x-api-key: $KEY" -OJ "$BASE/media/download?url=https://vimeo.com/76979871&ext=mp4"

Web app (fetch)

Call this from your own backend, not directly from the browser, so the API key never reaches client-side JS:

const res = await fetch(`${BASE}/media/info?url=${encodeURIComponent(url)}`, {
  headers: { "x-api-key": process.env.DLP_WRAPPER_KEY },
});
const info = await res.json();

Discord bot

A minimal discord.js example ships in examples/discord-bot. The core call looks like:

const r = await fetch(`${DLP_WRAPPER_BASE}/media/direct-url?url=${encodeURIComponent(url)}&format=best`, {
  headers: { "x-api-key": DLP_WRAPPER_KEY },
});
const { urls } = await r.json();
await message.reply(`Direct link: ${urls[0]}`);

Deploy — Render

The repo includes a render.yaml blueprint.

  1. Push this repo to GitHub
  2. On Render: New → Blueprint, connect the repo
  3. Render reads render.yaml and provisions the service — it generates API_KEY for you
  4. Set ALLOWED_ORIGINS to your web app's origin if it'll call the API from browser JS
  5. Deploy — your base URL is https://<service-name>.onrender.com/api
The default plan in render.yaml is sized for light use. Bump it if you expect concurrent transcodes — ffmpeg and yt-dlp are both CPU/RAM heavy.

Deploy — Docker

docker build -t dlp-wrapper .
docker run --rm -p 8080:8080 \
  -e PORT=8080 \
  -e API_KEY=your-secret-key \
  dlp-wrapper
